# NQ EOB — OR + EOB Backtest Capture Skill

## Overview

Backtesting/historical companion to `nq-snapshot`, for bang's **30-minute Opening Range + multi-timeframe EOB** model.

The model uses:

- **BT VWAP** (`/Users/nolan/Projects/Trading/Indicators/Code/BT_VWAP.pine`) to define the 30-minute OR (09:30–10:00 ET), OR mid, full-range targets, VWAP context.
- **EOB Advanced** (`/Users/nolan/Projects/Trading/Indicators/Code/BT_EOB_v2.pine`, on-chart name "EOB — Advanced Enhanced Order Block v3") for Turtle Soup / CISD / Continuation EOBs on 1m/2m/3m/5m with grade/score.
- **Long/Short Position tools** (LineToolRiskReward) drawn by bang on a replay chart as the trade markings.

The deliverable is always **rows in the Backtest Lab DB + day screenshot + raw context archive**, ready for bang to review/edit in the app.

## PRIMARY WORKFLOW — Replay Markup Capture

> Read **[references/replay-markup-capture.md](references/replay-markup-capture.md)** and follow it exactly. Validated 2026-06-07 (5/1 session, 7 trades extracted, merged, verified).

The loop bang wants:

1. bang marks T1–Tn with position tools on the 2m replay chart (frontier at EOD), states any per-trade TF exceptions, says "pull the day".
2. Claude: `replay_status` → extract drawings via **`ui_evaluate`** (NOT `draw_list` — known broken; stop/profit levels are **tick offsets × 0.25**) → pull 2m OHLCV (~300 bars) → `capture_screenshot` for the day-level image.
3. **Persist the bars** → write `data/<date>-nq1-2m.csv` to the vault AND `POST /api/bt/bars` so the session candles are queryable (the market-behavior layer; never discard them).
4. Compute the **OR from OHLCV** (09:30–10:00 ET; epochs are true UTC) and run the **deterministic stop/target walk** per trade → raw outcome, exit time, MFE/MAE in R (conservative stop-first; stop-at-the-wick entry-bar rule).
5. **UPSERT into the Backtest Lab** via the journal-app API (`/api/bt/...`) — match existing rows by entry ±2 pts or label; **bang's manual fields always win** (tp, notes, grade, bias, regime, managed, day note); inserts get `auto:`-prefixed notes and blank judgment fields.
6. **Recompute features** → `POST /api/bt/features/recompute?model_id={id}&date=<date>` snapshots each entry bar's VWAP / dist-from-VWAP-in-R / EMA stack / ATR / OR-position / Keltner into the trade's `features` JSON (from the persisted bars).
7. Attach the day screenshot, archive raw extraction JSON to the vault model folder, and report the per-trade table + review checklist.
8. bang refreshes the app, edits, hits **Commit → vault**, advances replay to the next day.

## Model Definition (unchanged)

- **OR**: first 30 min of RTH (09:30–10:00 ET) per BT VWAP (`or_top`, `or_bot`, `or_mid`, full-range targets = ±1 OR range).
- **EOB sequence**: Candle 1 (the EOB) → Candle 2 (displacement engulf) → Candle 3 (retrace/tap = entry). Types: TS / CISD / CONT. EOB Advanced adds grade/score/regime/session/HTF-bias.
- bang's rulebook lives in the app's Rules card and exports to `30m-or-eob-rules.md` — read it for current entry/risk/TP rules before classifying.

## Canonical Setup Codes

| Setup code | Meaning | Direction | First target |
|---|---|---|---|
| `OR-EOB-Fail-High` | Failed break above OR high; bearish EOB at/above upper edge | Short back into range | OR mid → OR low |
| `OR-EOB-Fail-Low` | Failed break below OR low; bullish EOB at/below lower edge | Long back into range | OR mid → OR high |
| `OR-EOB-Cont-Up` | Accepted break above OR; bullish continuation EOB on backtest | Long continuation | Prior high / extension / liquidity |
| `OR-EOB-Cont-Down` | Accepted break below OR; bearish continuation EOB on backtest | Short continuation | Prior low / extension / liquidity |
| `EOB-PreOR-Data` | Pre-OR trade (e.g. 08:30 ET data print) — OR doesn't exist yet | either | per drawing |

Never merge reversal and continuation stats; never force OR codes onto pre-OR trades.

## Critical Rules

0. **Verify the live model id before ANY write.** `SELECT id, slug FROM bt_models` (or `GET /api/bt/models`). The roster changes while you're away — bang deleted `30m-or-eob` (id=2) on 2026-07-05 and its trades cascaded away; `2m-kspan-eob` (id=3) became the active model. Never hardcode a model id from a prior session or from this file; never recreate a deleted model unprompted.
1. **Drawings via `ui_evaluate`, not `draw_list`** (known `getChartApi` bug). Decode stop/profit as tick offsets × 0.25.
2. **Merge, never clobber.** bang's tp / notes / grade / bias / regime / managed / day note always survive an upsert. Machine notes are prefixed `auto:`.
3. **State the OR source** (`computed_ohlcv` / `bt_vwap` / `screenshot_estimate` / `manual`).
4. **Preserve EOB timeframe per trade** (default 2m; bang states exceptions like "T4 is 3m").
5. **Walk conventions are explicit**: conservative stop-first; stop-at-the-wick entry-bar rule; bang's managed outcome is truth, the walk is the raw-mechanical baseline.
6. **Persist + derive, don't discard**: always store the 2m bars (`bt_bars` + CSV), then recompute per-trade features from them. Live indicator panes can't be seeked per-bar in replay, but VWAP/EMA/ATR/OR/Keltner are re-derived deterministically from OHLCV — so per-trade features ARE available post-hoc. Only bespoke Pine studies not derivable from price are unavailable; say so when relevant.
7. **Durable outputs in the app DB + vault**, nothing ad hoc.

## TradingView MCP field notes (validated 2026-07-05)

- **CDP is pinned to ONE tab.** `tab_switch` succeeds but `ui_evaluate`/`data_get_ohlcv` keep talking to the originally-attached tab (check `location.href` inside an eval if unsure). To reach a chart in another layout, use **`layout_switch`** on the attached tab — do NOT trust tab_switch for extraction.
- **Drawings sync per SYMBOL, not per layout.** All layouts show the same shape set for a symbol. If position tools aren't on MNQ1!, check NQ1! (and vice versa) before concluding the day isn't marked. Scan every pane: `TradingViewApi.chartsCount()` + `chart(i).getAllShapes()`.
- **No marked day found ≠ stall.** Prior markings are cleaned up after capture. Fall back to the highest-value unambiguous work: backfill `bt_bars` for any DB day missing bars (trades with blank features are the signal — `skipped_no_bars` in the recompute response), then ask bang to mark the next day.
- **`replay_start date=D` sets the frontier to D−1 23:59:59** (start-of-D minus 1s). So to pull day D's full session, `replay_start` at **D+1**. `data_get_ohlcv count=500` returns ~300 bars ≈ 09:00–19:58 ET with the 17:00–18:00 maintenance gap — full RTH coverage.
- **Bar integrity checks before persisting**: exact bar count matches the pull, times strictly increasing, only the 3720s maintenance gap, OHLC sanity (l ≤ o,c ≤ h). Then CSV to vault + `POST /api/bt/bars` + verify `GET /api/bt/bars/dates`.
- **Restore bang's replay state when done** (note the frontier from `replay_status` BEFORE touching replay; restore with `replay_start` at the nearest date boundary and tell him what moved).
- **Cross-check trade prices against the bars' session range** after any pull — entries outside [session_low, session_high] mean wrong symbol/date/instrument somewhere; flag, don't silently persist derived values.

## App-side automation (don't duplicate work)

The Journal app now auto-computes **MFE/MAE from stored bars when both are blank** on save, and `POST /api/bt/mfe/recompute?model_id=` backfills. The capture walk should still write its computed values explicitly (they're part of the raw archive), but expect the app to fill blanks on bang's manual rows. `setup_tag` options come from `bt_models.setup_tags` — use tags from that list when classifying.
