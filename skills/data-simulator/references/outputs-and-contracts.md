# Outputs & Contracts — the four layers

All four derive from the same ledger/config. Layer order matters: bronze and vendor files first, database from bronze, contract from gold.

## Layer 1 — Vendor-exact source files (`data/sources/<vendor>/`)

One folder per vendor, files named by the schema YAML `filename_pattern`, columns in exact YAML order, quirks applied (comment blocks, title/total rows, currency/percent strings, blank continuation rows, tab-delimited settlement files, etc.). Formats: CSV (default), TXT (tab-delim where the schema says), XLSX **with live formulas** for internal docs (openpyxl — formulas make unstructured-ingest demos honest).

Purpose: pipeline/medallion demos (Fabric, Databricks, Snowflake, dbt) and "we understand your stack" credibility in the room.

## Layer 2 — Idealized bronze CSVs (`data/bronze/`)

The 9-domain medallion starter (adapt domains to the business model — see source-taxonomy):

| domain | file | grain |
|---|---|---|
| sales | sales_daily.csv | date × channel × sku (+ dirt) |
| marketing | marketing_spend_daily.csv | date × channel × platform |
| marketing | web_traffic_daily.csv | date × channel |
| marketing | email_sms_daily.csv | date × program |
| inventory | inventory_snapshot.csv | week × sku × DC |
| retention | retention_cohorts.csv | cohort_month × months_since |
| creators | creators.csv | month × creator |
| campaigns | campaigns.csv | campaign |
| aivisibility | ai_visibility_weekly.csv | week × platform × query × brand |
| launch | launch_scorecard.csv | launch × milestone × metric |
| reference | dim_product / dim_channel / events_calendar | — |

Core fact spec — `sales_daily`: `order_date, channel_id, sku, units, gross_revenue, discount, returns, net_revenue, orders`; identity `net = gross − discount − returns` on all clean rows. Other domain shapes follow the grains above; document exact columns in the run's DATA_DICTIONARY.md.

## Layer 3 — Database (`data/{slug}.duckdb`, optional Supabase)

- `bronze.*` — CSVs loaded raw (`read_csv_auto`), dirt intact
- `silver.*` — cleaned: trim/upper keys, coalesce nulls, dedupe, event-flag join, typed columns
- `gold.*` — star schema: `fact_sales`, `fact_marketing`, `dim_date`, `dim_product`, `dim_channel`, plus per-capability marts (`mart_exec_monthly`, `mart_retail_weekly`, `mart_creator_monthly`, `mart_ecom_daily`)

Supabase (only when the demo needs hosted data): push `gold.*` into schema `sim_{slug}` via the Supabase MCP (`apply_migration` for DDL, batched `execute_sql` inserts, or CSV via storage). Business data NEVER goes in `vail_*` — that schema is access-gating/progress only. Keys from env (`WL_SUPA_URL`, `WL_SUPA_KEY`); never copy from reference projects.

## Layer 4 — Growth Desk contract (`contract/`)

Generated by `export_contract.py` querying **gold** — never hand-typed, so demo numbers cannot drift from the ledger. (Earlier engagements kept the generator ladder and the demo's JS anchors in sync by hand; deriving both from one config is the fix.)

### `growth-desk-data.js`

ES module of plain objects matching the Growth Desk client shapes. Illustrative shape only — every key and value derives from the active client's gold tables (channel keys from `dim_channel`, categories from `dim_product`):

```js
export const months = [           // one per month, trailing 12–18 shown in demo
  { month: 'Jan', sales: <net $M>, plan: <plan $M>, retail: <..>, dtc: <..>,
    <channelKey1>: <..>, <channelKey2>: <..>,   // camelCase ids derived from dim_channel
    spend: <..>, roas: <..>,
    categories: { <Category1>: <..>, <Category2>: <..> },   // from dim_product categories
    note: 'auto-generated: <driver of the month>' },
  ...
];
export const channels = [...];    // id, name, share, yoy, aov
export const products = [...];    // sku, name, category, price, units, revenue, margin
export const campaigns = [...];
export const socialPlatforms = [...];
export const creators = [...];    // handle, platform, tier, engagement, attributed_revenue
export const competitors = [...]; // from dossier competitor set
export const RETAILERS = [...];   // id, name, share, drift, aur, kind, color
export const weeks = [...];       // { week, <retailerId>: { pos, shipped } } — weekly grain, full window
```

Channel keys inside `months[]` are brand-specific — derive from `dim_channel`, keep camelCase ids stable across all contract files. Weekly series must use day-accrual weighting across month boundaries so weekly YoY comps stay clean over 4/5-week months.

### `agent-context/{capability}.json`

One JSON blob per demo agent capability — the `context` passed to `POST /api/agent`. The broker's system prompt forbids the model from contradicting this context, so it must be self-sufficient. Default Growth Desk capabilities (mirror the demo repo's `lib/agent/broker.ts`; extend if the demo adds tools):

| capability | broker tool | context contents (from gold) |
|---|---|---|
| ecom_replenishment | propose_replenishment_plan | sku velocity, on-hand, weeks-cover, stockout risks |
| ecom_counter_move | propose_counter_move | daily sales vs plan, competitor signals, live promos |
| creator_shortlist | build_creator_shortlist | creator roster w/ tiers, engagement, attributed revenue |
| creator_spend_match | match_creator_spend | creator spend/revenue, budget remaining |
| campaign_content | generate_campaign_content | campaign calendar, category performance, brand moments |
| exec_brief | propose_executive_readout | mart_exec_monthly: sales/plan/mix/roas + notes |

Validate every JSON file parses and every number in it exists in gold (spot-check queries).
